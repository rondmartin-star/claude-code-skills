## Decision Framework

### ✅ When to Add Parallelization

**Criteria (ALL must be true):**
1. Operations are **independent** (no dependencies)
2. Commonly processes **10+ items**
3. Expected **speedup > 2x**
4. **Production use case** involves batch operations

**Examples:**
- Batch CRUD operations (document-management)
- Multi-platform publishing (content-creation)
- Parallel validation (audit-orchestrator)
- Concurrent analysis (convergence-engine)

### ❌ When to Skip Parallelization

**Criteria (ANY is true):**
1. **Sequential workflow** (step A before step B)
2. **Single-item operations** (one at a time)
3. **User interaction** (human in the loop)
4. **Data integrity** (order matters)
5. **Planning/design** (inherently sequential)

**Examples:**
- Status checks (corpus-detect)
- Configuration edits (corpus-config)
- User confirmation (confirm-operation)
- Sequential setup (corpus-init)
- Planning activities (battle-plan Phase 1)

**What it would have been:**
- 61/61 skills parallelized (100%)
- Even sequential operations forced
- 7+ hours of work
- Added complexity with no benefit
- Potential bugs from race conditions

**Why we rejected it:**
- Parallelizing sequential workflows = bugs
- Parallelizing single-item ops = overhead > benefit
- Parallelizing user interaction = impossible
- Forced parallelization = wrong approach

- 24/61 skills parallelized (39%)

- 4 hours of smart work (vs 7+ hours forcing)

- Production-validated approach

**Result:** Right level of parallelization for the ecosystem
